**Revision:** 1 · **Last modified:** 2026-07-01T00:00:00Z

Per-feature status ledger (§11.4.153). Every user-facing capability of the
HelixCode stack, its test-type coverage (§11.4.169), validation verdict, and
the captured-evidence path that backs it. Validation vocabulary: PASS / SKIP /
OPERATOR-BLOCKED / PENDING (§11.4.45). Evidence lives under the git-ignored
`qa-results/` tree (regenerated by re-running the suite, §11.4.77); curated
copies for this release are under `docs/qa/<run-id>/` (§11.4.83).

## Components
- **Caddy edge** — TLS termination + HTTP→HTTPS redirect + reverse proxy (host 52443/52080).
- **code-server** — browser VS Code, auth-gated, projects bind-mounted RW.
- **Operator scripts** — install / setup / start / stop / restart / status / logs / doctor / set-password / tune-host / (un)install-service.
- **Boot-survival service** — rootless systemd user unit (linger, WantedBy=default.target).

## Honest boundaries (§11.4.6)
- Real, publicly-trusted Let's Encrypt cannot be autonomously proven on a LAN box
  (no public domain); the ACME issue/serve/rotate code path is proven against a
  local Pebble CA, and the operator enables the public CA per `docs/guides/TLS.md`.
- Full in-browser editor interaction (typing, file tree, command palette) is
  `operator_attended` until a browser-automation adapter is wired (HC-QA-UI-001);
  every HTTP-drivable journey is autonomously verified.
